# RCT v1.0.0 Implementation Plan

**Date:** 2026-03-31
**Spec:** `docs/specs/2026-03-31-rct-v1.0.0-design.md`
**Structure:** 3 slices (A → B → C), TDD execution, Haiku review gates

## Execution Framework

Each workstream within a slice follows this cycle:

1. **Write failing tests** — subagent writes tests against the spec before any implementation
2. **Implement** — subagent implements until all tests pass
3. **Spec compliance audit** — Haiku agent verifies implementation matches spec
4. **Code quality review** — Haiku agent reviews for anti-patterns, dead code, style
5. **Fix cycle** — address review findings, re-run reviews until clean
6. **Accept** — workstream merged to working branch

Independent workstreams within a slice run in parallel. Cross-slice dependencies:
- Slice A blocks Slice B
- Slice A blocks C1, C2 (lib surface and plugin resolve can start after A, in parallel with B)
- C3 (plugin discovery) depends on B5 (init wizard) + C2 (plugin resolve)
- C4 (JSDoc) depends on C1
- C5 (docs) depends on all other steps

### C2.5: meta.ts plugin source fix

**Problem:** `getPluginSectionValues()` in `meta.ts` reads from the hardcoded built-in registry, not from the resolved config. After C2 introduces dynamic plugin resolution, local/package plugins' contributions are merged into `config.files`/`config.rules` by `applyPlugins()`, but `meta.ts` re-reads the built-in registry separately — double-counting built-ins and missing local/package plugins.

**Files to modify:**
- `src/engine/meta.ts` — remove `getPluginSectionValues()`. The `buildFilesSection()` and `buildRulesSection()` functions already receive the config that `applyPlugins()` has merged plugin contributions into. They should read from `config.files` and `config.rules` directly (which already contain plugin contributions), not re-read from the registry.

**Tests:**
- `test/meta.test.ts` — add test: meta output for a config with a local plugin includes that plugin's files/rules in the enumeration
- `test/meta.test.ts` — verify no double-counting of built-in plugin contributions

**Dependency:** C2 (plugin resolve, so `applyPlugins()` handles all plugin types)
