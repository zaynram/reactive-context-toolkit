# RCT v1.0.0 Design Specification

**Date:** 2026-03-31
**Status:** Reviewed (plan-review protocol applied; 10 findings addressed)
**Supersedes:** 2026-03-28-rct-v1-design.md (pre-implementation sketch)

## 1. Goals

Ship reactive-context-toolkit v1.0.0 with production quality:

1. **Simplify** — remove dead code, consolidate abstractions, eliminate anti-patterns
2. **Restructure** — clarify the dual-use nature (CLI binary + library for extension)
3. **Fix bugs** — settings generation, node detection false positives
4. **Scope tests per language** — granular test configuration for multi-language projects
5. **Formalize plugins** — typed authoring API, discovery for local + external plugins
6. **Derive-then-persist config** — interactive init, non-interactive update, explicit defaults

## 1.1 Non-goals (deferred to post-v1.0.0)

- Extracting built-in plugins to separate repos/packages (premature; no external plugin authors yet)
- Automatic `rct update` merge without preview (needs more UX validation)
- Version-pinned plugin references (`rct-plugin-foo@^1.0`)

## 1.2 Build and distribution model

RCT is installed from GitHub via `bun add github:zaynram/reactive-context-toolkit`. Bun resolves the `bin.rct` field directly to `src/cli/index.ts` — there is **no build step and no `dist/` directory**. The hook command is `bun run rct hook <event>`, which executes source directly. All references to `dist/hook.js` or `dist/rct.js` in docs are legacy and should be removed.

## 2. Breaking Changes

### 2.1 SupportedLanguage

```typescript
// v0.x
export const LANGUAGES = ['python', 'javascript', 'typescript', 'rust'] as const
type SupportedLanguage = 'python' | 'javascript' | 'typescript' | 'rust'

// v1.0.0
export const LANGUAGES = ['node', 'python', 'rust'] as const
type SupportedLanguage = 'node' | 'python' | 'rust'
```

`LangConfig` changes from `{ typescript?, javascript?, python?, rust? }` to `{ node?, python?, rust? }`. The `node` language encompasses both TypeScript and JavaScript projects. Config files (tsconfig.json, jsconfig.json) are grouped under `lang.node.config`.

**Migration:** `validateConfig()` detects legacy `typescript`/`javascript` keys and emits a warning with migration instructions: `"lang.typescript is deprecated in v1.0.0. Rename to lang.node."` The legacy keys are not silently ignored — the warning is actionable.

### 2.2 Plugin config format

```json
// v0.x — built-in names only
{ "globals": { "plugins": ["track-work", "issue-scope"] } }

// v1.0.0 — built-in names, local files, or external packages
{ "globals": { "plugins": ["track-work", ".claude/hooks/rct/my-rules.ts", "@myorg/rct-plugin-standards"] } }
```

Built-in plugins remain in-tree (not externalized). They are resolved by name from the hardcoded registry. Local and package plugins use dynamic `import()`.

### 2.3 Per-language test config

`LangEntry` gains an optional `test` field. Top-level `TestConfig` provides shared defaults that are inherited per-language (not an all-or-nothing fallback).

### 2.4 CLI commands

- `rct init` becomes an interactive wizard (was non-interactive). Falls back to non-interactive mode when `!process.stdin.isTTY` or `--yes` flag is passed.
- `rct update` is new (non-interactive re-derive + merge with derivation baseline)
- `rct hook <event>` unchanged

## 4. Detailed Design

### 4.1 Language module restoration (lang/)

Each `lang/*.ts` file becomes a per-language aggregator that owns the full evaluation logic for its ecosystem.

**`lang/node.ts`:**

- Imports `getBunScripts`, `getBunWorkspace` from `#tools/bun`, and npm/pnpm tool defs
- Exports `evaluateNode(entry: LangEntry, event: HookEvent, cwd: string): string[]`
- Contains `extractTsconfigPaths()` (moved from `lang/index.ts`)
- Auto-discovers `tsconfig.json` / `jsconfig.json` if `entry.config` is omitted

**`lang/python.ts`:**

- Imports `getPixiTasks`, `getPixiEnvironment` from `#tools/pixi`
- Exports `evaluatePython(entry: LangEntry, event: HookEvent, cwd: string): string[]`

**`lang/rust.ts`:**

- Imports `getCargoInfo` from `#tools/cargo`
- Exports `evaluateRust(entry: LangEntry, event: HookEvent, cwd: string): string[]`

**`lang/index.ts`:**

- `evaluateLang(lang: LangConfig, event: HookEvent, cwd: string): string[]`
- Dispatches: `lang.node` → `evaluateNode()`, `lang.python` → `evaluatePython()`, `lang.rust` → `evaluateRust()`
- Drops the unused `_globals` parameter

**Config auto-discovery pattern** (in each lang module — operates on a cloned entry, not the original):

```typescript
// Clone entry to avoid mutating the config object
const resolved = { ...entry }
if (!resolved.config) {
    if (fs.exists(fs.join(cwd, 'tsconfig.json'))) {
        resolved.config = [
            { name: 'tsconfig', path: 'tsconfig.json', extractPaths: true },
        ]
    } else if (fs.exists(fs.join(cwd, 'jsconfig.json'))) {
        resolved.config = [
            { name: 'jsconfig', path: 'jsconfig.json', extractPaths: true },
        ]
    }
}
```

### 4.2 Config derivation model

**Core principle:** every config field has a derivation function. `rct init` and `rct update` use these to produce fully-populated configs.

**New file: `config/derive.ts`**

```typescript
interface DerivedConfig {
    lang: LangConfig
    test: TestConfig | null
    files: FileEntry[]
    globals: Partial<GlobalsConfig>
}

export function deriveFromProject(root: string): DerivedConfig
```

This replaces the current `detectProject()` + `generateConfig()` pair. It returns a fully-populated config with all derived values explicit.

**Derivation sources:**

- Language detection: tsconfig.json, jsconfig.json, package.json (relaxed gate), pixi.toml, pyproject.toml, Cargo.toml
- Package manager: bun.lock → bun, pnpm-lock.yaml → pnpm, package-lock.json → npm
- Test commands: from package.json scripts, pixi tasks, or tool defaults
- Config files: auto-discovered per language

**Node detection gate (relaxed per plan-review F6):**

```typescript
const pkg = hasPkg ? JSON.parse(fs.readRaw(pkgPath)) : null
const isNodeProject =
    hasTsconfig ||
    hasJsconfig ||
    (pkg?.name && (pkg?.version || pkg?.private || pkg?.scripts))
```

A `package.json` with `name` and at least one of (`version`, `private`, `scripts`) is treated as a Node project. A `package.json` with only `dependencies` (e.g., an RCT consumer's minimal manifest) is not.

### 4.3 CLI commands

#### `rct init` — Interactive wizard

Uses `cli/prompt.ts` utilities (`ask()`, `confirm()`, `select()`). Hand-rolled minimal readline wrapper (~50 lines), no external dependencies.

**TTY detection (per plan-review F3):** When `!process.stdin.isTTY` or `--yes` is passed, all prompts are skipped and derived defaults are used. This preserves compatibility with CI, Docker, and automated setup scripts.

Flow:

1. Run `deriveFromProject()` to get defaults
2. If non-interactive: write config with derived defaults, skip to step 8
3. Ask questions with derived values as defaults (Enter to accept)
4. Language selection (multi-select from detected + manual add)
5. Per-language: package manager, test command
6. Plugin selection (multi-select: built-in + discovered local plugins)
7. Output format (xml/json), test caching (yes/no)
8. Write `rct.config.json` with all values explicit
9. Run `mergeSettings()` on fully-resolved config (after `applyPlugins()` + `desugarFileInjections()`) to update `.claude/settings.json`

Re-running on existing config warns and asks to overwrite or abort (in interactive mode) or aborts with a message (in non-interactive mode).

#### `rct update` — Non-interactive re-derive + merge

Uses a **three-way merge** with a stored derivation baseline.

1. Load existing `rct.config.json` (includes `_derived` baseline from last init/update)
2. Run `deriveFromProject()` for fresh derivation
3. Three-way merge for each config field:
    - If `current == baseline` → field was auto-generated, safe to update with fresh value
    - If `current != baseline` → consumer customized it → preserve consumer's value
    - If `baseline` has a field that fresh derivation doesn't → project marker removed → remove field (unless consumer modified it)
    - If fresh derivation has a field that `baseline` doesn't → new detection → add it
4. Write updated config with new `_derived` baseline
5. Run `mergeSettings()` on fully-resolved config to update `.claude/settings.json`

**`_derived` key in config:**

```json
{
    "lang": { "node": { ... } },
    "test": { ... },
    "_derived": {
        "lang": { "node": { ... } },
        "test": { ... }
    }
}
```

The `_derived` key stores the last derivation output. It is written by `rct init` and `rct update`, ignored by the evaluation pipeline, and stripped from schema validation. Merge granularity is per top-level field (`lang`, `test`, `files`, `globals`). Per-property merge within objects (e.g., individual tool entries) is deferred — if any part of a section differs from baseline, the whole section is treated as consumer-customized.

#### `rct hook <event>` — Unchanged

Pipeline is the same. `applyPlugins()` becomes async (for dynamic plugin imports). `evaluateLang()` drops unused `_globals` parameter.

### 4.4 Settings generation fix

`mergeSettings()` is rewritten to collect required hook events from ALL config sections.

**Ordering (per plan-review F9):** `mergeSettings()` operates on the fully-resolved config — after `applyPlugins()` and `desugarFileInjections()` have run. This ensures plugin-contributed rules/injections/files are included in the event scan.

```typescript
function collectRequiredEvents(config: RCTConfig): Set<HookEvent> {
    const events = new Set<HookEvent>(['SessionStart']) // always needed

    for (const rule of config.rules ?? []) events.add(rule.on)
    for (const inj of config.injections ?? []) events.add(inj.on)
    for (const file of config.files ?? []) {
        const on = file.injectOn
        if (on) (Array.isArray(on) ? on : [on]).forEach((e) => events.add(e))
    }
    if (config.lang) {
        for (const entry of Object.values(config.lang)) {
            if (!entry) continue
            const on = entry.injectOn
            if (on)
                (Array.isArray(on) ? on : [on]).forEach((e) => events.add(e))
            for (const tool of entry.tools ?? []) {
                const ton = tool.injectOn
                if (ton)
                    (Array.isArray(ton) ? ton : [ton]).forEach((e) =>
                        events.add(e),
                    )
            }
        }
    }
    if (config.test) {
        const tc = typeof config.test === 'object' ? config.test : {}
        const on = tc.injectOn
        if (on) (Array.isArray(on) ? on : [on]).forEach((e) => events.add(e))
    }
    if (config.meta) {
        const on = config.meta.injectOn
        if (on) (Array.isArray(on) ? on : [on]).forEach((e) => events.add(e))
    }

    return events
}
```

For each event, generate a settings hook entry. For `PreToolUse`/`PostToolUse`, also collect matchers from rules and injections.

### 4.5 Per-language test scoping

**New type:**

```typescript
interface LangTestConfig {
    command?: string | true // true = auto-detect from tools
    brief?: string // template: "{status}", "{lang}: {status} ({tool})"
    format?: Format // override global format
}
```

**Updated `LangEntry`:**

```typescript
interface LangEntry {
    tools?: LangTool[]
    config?: LangConfigFile[]
    injectOn?: HookEventOrArray
    test?: LangTestConfig // NEW
}
```

**Top-level `TestConfig`** provides shared defaults: `injectOn`, `cache`, `cacheTTL`, and serves as a per-language fallback (not all-or-nothing).

**Execution in `hook.ts` (per plan-review F4):**

```typescript
const topLevelTest = resolveTopLevelTestConfig(desugared.test)

for (const [langName, entry] of Object.entries(desugared.lang ?? {})) {
    if (!entry) continue
    // Per-language test inherits from top-level if not specified
    const langTest = entry.test ?? topLevelTest
    if (!langTest) continue

    const command = resolveTestCommand(langTest, entry)
    if (!command) continue
    const result = runTest(command.command, cwd)
    testResults.push(
        formatTestResult(
            { ...result, tool: command.tool, lang: langName },
            langTest,
            topLevelTest,
            globals,
        ),
    )
}

// If no languages are configured but top-level test exists, run it standalone
if (testResults.length === 0 && topLevelTest) {
    // existing v0.x behavior as fallback
}
```

This ensures that a project with `lang.node.test` set and `lang.rust` without a `test` field still runs the top-level test config for Rust (inherits `entry.test ?? topLevelTest`).

**Output format:**

```xml
<test lang="node" tool="bun" status="pass"/>
<test lang="rust" tool="cargo" status="fail" exitCode="1"/>
```

**Brief templates** support placeholders: `{status}`, `{exitCode}`, `{output}`, `{tool}`, `{lang}`.

### 4.6 Plugin system

#### Plugin interface

```typescript
interface RCTPlugin {
    name: string
    files?: FileEntry[]
    rules?: RuleEntry[]
}
```

Unchanged from v0.x. The `definePlugin()` helper validates shape at compile time:

```typescript
// lib/plugin.ts
export function definePlugin(plugin: RCTPlugin): RCTPlugin {
    return plugin
}
```

#### Built-in plugins (stay in-tree)

`src/plugin/issueScope.ts` and `src/plugin/trackWork.ts` remain in the core package. They switch from hardcoded `node_modules/...` paths to `fs.source()` for schema resolution. This is the simplest approach that works today; externalization to separate packages is deferred until there is demonstrated external plugin demand.

#### Plugin resolution (`plugin/resolve.ts`)

```typescript
type PluginSource = 'builtin' | 'local' | 'package'

interface ResolvedPlugin {
    plugin: RCTPlugin
    source: PluginSource
    ref: string // original string from config
}

export async function resolvePlugin(ref: string): Promise<ResolvedPlugin>
```

Resolution chain:

1. Matches a built-in name (`track-work`, `issue-scope`) → **built-in**. Resolved from the inline registry. No `import()`.
2. Starts with `.` or `/` → **local file**. Resolved relative to `CLAUDE_PROJECT_DIR`. Convention: `.claude/hooks/rct/*.{ts,js}`. Dynamic `import()`, expect default export satisfying `RCTPlugin`. Validation: if the import result doesn't have a `name` property, throw with a clear error message.
3. Otherwise → **package**. Dynamic `import(ref)`, expect default export satisfying `RCTPlugin`. If `import()` throws `MODULE_NOT_FOUND`, emit a clear error: `"Plugin '${ref}' not found. Install it with: bun add ${ref}"`.

#### Plugin discovery for wizard

During `rct init`:

1. List built-in plugins by name from the inline registry
2. Scan `.claude/hooks/rct/*.{ts,js}` for local plugin files
3. Scan `node_modules/rct-plugin-*` for installed plugin packages
4. Present all discovered plugins in multi-select

#### `applyPlugins()` becomes async

```typescript
// Before (sync, hardcoded registry lookup)
export function applyPlugins(config: RCTConfig): RCTConfig

// After (async, dynamic resolution)
export async function applyPlugins(config: RCTConfig): Promise<RCTConfig>
```

All callers of `applyPlugins()` are already in async contexts (`cli/hook.ts` main(), `cli/init.ts`). Test files that call it synchronously will need to be updated to `await`.

### 4.7 Library surface (`src/lib/`)

#### `lib/plugin.ts` — Plugin authoring

```typescript
export function definePlugin(plugin: RCTPlugin): RCTPlugin
```

Typed identity function. Validates shape at compile time. Resolves relative paths against `process.cwd()`.

#### `lib/hook.ts` — Managed hook lifecycle

```typescript
type HookHandler = (
    input: RC.HookInput,
) => RC.HookJSONOutput | Promise<RC.HookJSONOutput>

export function createHook(handler: HookHandler): void
```

Wraps stdin parsing, handler invocation, stdout formatting, and exit code handling into a single call. Consumer writes a handler function, RCT manages the I/O.

#### `lib/register.ts` — Low-level hook I/O (advanced use)

Moved from `src/register.ts`. Anti-patterns fixed:

- `dynamic()`: removes dead `reject(block(...))` — `block()` calls `process.exit(2)` so reject never propagates. New implementation handles parse errors by writing an error JSON to stdout and exiting, without using `block()` inside a Promise rejection path.
- `standard()`: signature simplified where possible.
- `block()`: unchanged.

#### JSDoc on barrel exports

All exports from `src/index.ts` get JSDoc annotations documenting:

- What the function does
- When a consumer would use it
- For engine/config functions: that they can be composed for custom pipeline-access patterns (documents the advanced extension possibility without formalizing it as a separate API)

### 4.8 Dead code removal

| Target                               | Action                                                           |
| ------------------------------------ | ---------------------------------------------------------------- |
| `fs.config()`                        | Remove — zero usages in codebase                                 |
| `plugin/public.ts`                   | Delete — `getSchemaPath()` and `createFromTemplate()` are unused |
| `_globals` param on `evaluateLang()` | Remove                                                           |
| `reject(block())` in `dynamic()`     | Fix anti-pattern                                                 |
| `build` script in `package.json`     | Remove — no build step; Bun resolves bin to source directly      |
| `dist/` references in docs           | Remove — no build artifacts exist                                |

Note: `src/tools/clippy.ts` and `src/tools/ruff.ts` are already deleted in the working tree. No action needed.

### 4.9 node_modules path elimination

Plugin schema paths in `issueScope.ts` and `trackWork.ts` switch from hardcoded `node_modules/reactive-context-toolkit/public/schema/...` to `fs.source(['public', 'schema', name])`. The `fs.source()` helper resolves from `RCT_PREFIX`, which is already set correctly.

`RCT_PREFIX` in `fs.ts` remains for resolving core assets (schemas, templates). It is not used for plugin resolution (which goes through `plugin/resolve.ts`).

## 5. Schema Changes

### 5.1 `rct.config.schema.json` updates

- `LangConfig` properties: replace `typescript`, `javascript` with `node`
- `LangEntry`: add optional `test` property (`LangTestConfig`)
- `LangTestConfig`: `{ command?: string | true, brief?: string, format?: "xml" | "json" }`

- `SupportedLanguage` enum: `["node", "python", "rust"]`
- `_derived`: optional top-level key (stripped from validation, used by `rct update` for merge baseline)

### 5.2 `HookEvent` — unchanged

`SessionStart`, `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `UserPromptSubmit`, `SubagentStart`, `Notification`, `Setup`

## 6. Execution Plan Structure

### Slice A — Type system + cleanup

**Scope:** Workstreams 1, 2, 4
**Dependencies:** None
**Deliverables:**

- `SupportedLanguage` type change (`node`/`python`/`rust`)
- `lang/` module restoration (node.ts, python.ts, rust.ts) with auto-discovery (cloned entries)
- Dead code removal (fs.config, public.ts, \_globals, build script, dist/ references)
- `node_modules` path elimination in plugin definitions → `fs.source()`
- `register.ts` anti-pattern fixes + move to `lib/register.ts`

### Slice B — Config & init

**Scope:** Workstreams 5, 6
**Dependencies:** Slice A (needs new type system and lang/ modules)
**Deliverables:**

- `config/derive.ts` with full project derivation (relaxed node detection gate)

- Per-language `LangTestConfig` on `LangEntry` with top-level inheritance

### Slice C — Library surface + plugins

**Scope:** Workstream 3 + plugin formalization
**Dependencies:** Slice B (needs stable config model and init system)
**Deliverables:**

- `src/lib/` with `definePlugin()`, `createHook()`, `standard`/`dynamic`/`block`
- `plugin/resolve.ts` with resolution chain (built-in → local → package)

- `applyPlugins()` async migration
- JSDoc on all barrel exports
- CLAUDE.md + README.md final update
- `rct.config.schema.json` finalized

### Execution framework

Each slice uses TDD subagent delegation:

1. Subagent writes failing tests for the workstream
2. Subagent implements until all tests pass
3. Haiku agent runs spec compliance audit on changes
4. Haiku agent runs code quality review on changes
5. Fix/review cycle until clean
6. Accept workstream implementation

Independent workstreams within a slice run in parallel. Cross-slice dependencies enforce blockers (A blocks B, B blocks C).
