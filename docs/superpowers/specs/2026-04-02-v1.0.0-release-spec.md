## 1. Release Identity

Ship reactive-context-toolkit v1.0.0 as a stable, ship-and-forget release. The primary consumer is the DAISY project (stateful multi-hypothesis reasoning kernel). The developer intends to stop active RCT development after this release.

**Validated by tradeoff-analysis:** Alternative B (Enhanced) — ship current capabilities + self-briefing + process enforcement documentation. Alternatives C (MCP config tools) and D (worktree scoping) were decisively eliminated (1.40+ weighted-score gap). See `docs/roadmap.md` for conditional future work.

## 2. Design Principles

These apply to all execution plans:

1. **Intent over implementation** — Preserve what each piece is trying to do; rewrite freely if a better pattern exists.
2. **Simplify before bloating** — Prefer deleting code over adding abstractions. Three similar lines beat a premature helper.
3. **Additive-only plugins** — Plugins contribute to the pipeline (files, rules, context, triggers, setup). They cannot transform other plugins' output. This boundary is deliberate and validated.
4. **Pure functions where possible** — `definePlugin` must be side-effect-free. Setup runs in `applyPlugins` with error isolation, not at import time.
5. **Package name is identity** — The `ref` string (from config or BUILTIN_PLUGINS) is the canonical identifier. `plugin.name` is optional display metadata.
6. **Zero-dependency core** — RCT core has no runtime dependencies. Plugin packages may have dependencies (e.g., tmux uses `@modelcontextprotocol/sdk`).

## 3. Architecture Changes

### 3.1 Plugin Interface (`RCTPlugin`)

```typescript
interface RCTPlugin {
    name?: string                    // Optional display name; derived from ref if absent
    files?: FileEntry[]              // Declarative file registrations
    rules?: RuleEntry[]              // Declarative block/warn rules
    context?: (event, input) => string | undefined | Promise<...>
    trigger?: (event, input) => PluginTriggerResult | undefined | Promise<...>
    setup?: () => void | Promise<void>  // Lifecycle: called in applyPlugins, error-isolated
}
```

**Changes from current:**
- `name` becomes optional (currently required by validate)
- `setup` added as interface property (currently a second arg to `definePlugin`)
- `setup` signature simplified: no `plugin` parameter (plugin can close over its own state)

### 3.2 `definePlugin(plugin: RCTPlugin): RCTPlugin`

Pure function. Responsibilities:
1. Resolve relative paths in `files[].path` and `files[].metaFiles[].path` via `process.cwd()`
2. Return the plugin object unchanged otherwise

**Removed behavior:**
- No longer accepts a second `setup` argument
- No longer calls setup during evaluation

**Why `process.cwd()` is correct:** Plugin `files[].path` values like `.claude/context/scope.xml` are relative to the consumer's project directory. When RCT loads a plugin (via `applyPlugins` → `resolvePlugin` → `import`), `process.cwd()` is the project root. Plugin-internal asset paths (templates, schemas) use `__dirname` or `import.meta.dir` which are already absolute.

### 3.3 `validatePlugin(plugin: unknown, ref: string)`

Full replacement for `src/plugin/validate.ts`:

```typescript
export function validatePlugin(
    plugin: unknown,
    ref: string,
): asserts plugin is RCTPlugin {
    if (!plugin || typeof plugin !== 'object') {
        throw new PluginValidationError(
            `Plugin '${ref}' must export an object.`,
        )
    }
    const p = plugin as Record<string, unknown>
    const check = (key: string, type: string) => {
        if (key in p && typeof p[key] !== type) {
            throw new PluginValidationError(
                `Plugin '${ref}' property '${key}' must be ${type}, got ${typeof p[key]}`,
            )
        }
    }
    check('name', 'string')
    check('context', 'function')
    check('trigger', 'function')
    check('setup', 'function')
    if ('files' in p && !Array.isArray(p.files)) {
        throw new PluginValidationError(`Plugin '${ref}' property 'files' must be an array`)
    }
    if ('rules' in p && !Array.isArray(p.rules)) {
        throw new PluginValidationError(`Plugin '${ref}' property 'rules' must be an array`)
    }
}
```

Does NOT require `name`. The `ref` parameter provides identity.

### 3.4 Plugin Registry (`src/plugin/index.ts`)

**Key by BUILTIN_PLUGINS package name**, not by `plugin.name`:

```typescript
const results = await Promise.allSettled(
    BUILTIN_PLUGINS.map(async (name) => {
        const { default: plugin } = await import(name)
        validatePlugin(plugin, name)
        return plugin
    }),
)

// Pair results with names BEFORE filtering to avoid index corruption
export default Object.fromEntries(
    results
        .map((r, i) => [BUILTIN_PLUGINS[i], r] as const)
        .filter(([, r]) => r.status === 'fulfilled')
        .map(([name, r]) => [
            name,
            { plugin: (r as PromiseFulfilledResult<RCTPlugin>).value, ref: name, source: 'builtin' },
        ])
) as BuiltinPlugins
```

Remove `todo: wire in` comment.

### 3.5 Display Name Derivation

Utility function (in `src/plugin/types.ts` or `src/plugin/resolve.ts`):

```typescript
function displayName(plugin: RCTPlugin, ref: string): string {
    const raw = plugin.name ?? ref
    return raw.replace(/^rct-plugin-/, '')
}
```

Used in logging, error messages, and extension name tracking. The `applyPlugins` function uses `displayName(plugin, name)` instead of `plugin.name ?? name`.

### 3.6 `applyPlugins` — Setup Lifecycle

After resolving a plugin and merging files/rules, call `plugin.setup?.()` with error isolation:

```typescript
if (plugin.setup) {
    try {
        await Promise.resolve(plugin.setup())
    } catch (err) {
        console.warn(`[rct] Warning: plugin '${displayName(plugin, name)}' setup failed: ${err instanceof Error ? err.message : String(err)}`)
    }
}
```

**Insertion point:** This block goes as the last step inside the `for (const name of pluginNames)` loop in `applyPlugins`, after file/rule merging and context/trigger registration (after current line ~136).

**Setup failure does not roll back file/rule merges.** This is acceptable because:
- `buildFileRegistry` handles missing files gracefully (file content resolves to empty/undefined)
- `evaluateInjections` skips unresolved FileRefs with a warning (line 50-78 of `injections.ts`)
- Setup failures are already warned; the pipeline continues with degraded capability, not a crash

Setup runs once per `applyPlugins` call (which is once per hook invocation). Plugins that need idempotent setup should guard internally.

### 3.7 Pipeline Order (confirmed correct)

The current pipeline order in `src/cli/hook.ts` is correct and matches documentation:

1. Load config → validate → apply plugins (with setup) → desugar → build registry
2. Read stdin (async events)
3. **Plugin triggers** (early exit on block)
4. **Static rules** (early exit on block)
5. **Injections**
6. **Plugin contexts**
7. Warn messages (static rules + plugin triggers)
8. Meta
9. Lang
10. Test
11. Compose output

### 3.8 Self-Briefing Injection

A new built-in file registered by the core (not a plugin) that injects on SessionStart. Contains a concise description of what RCT is and what the current config provides. This is the "Alternative B" experiment from the tradeoff-analysis.

**Implementation:** A generated string (not a static file) produced during the compose phase, describing:
- Active plugins and their capabilities
- Registered file aliases
- Active rules (count + summary)
- Available injections

This leverages the existing `meta` system. The self-briefing IS the meta injection with `include: ['files', 'rules', 'plugins']`. No new code needed — just ensure `meta` config defaults are sensible and document the pattern.

**Decision:** Rather than adding a new self-briefing system, document that `meta` with appropriate `include` fields IS the self-briefing. The existing `generateMeta` function already produces this output. Add a note to CLAUDE.md and README recommending `meta` configuration for self-briefing.

## 4. Plugin Package Changes

### 4.1 Issue-Scope Plugin (`plugins/rct-plugin-issue-scope`)

**Current bug:** `setup()` crashes on null `metaFiles` and null `template.find()` result.

**Redesign:** Move setup to the `RCTPlugin.setup` property. Use closure pattern (not `this`) to access resolved files. Guard all optional access:

```typescript
const files: FileEntry[] = [
    {
        alias: 'scope',
        path: '.claude/context/scope.xml',
        metaFiles: scopeFiles,
        injectOn: 'SessionStart',
        staleCheck: { dateTag: 'date', wrapTag: 'stale-scope' },
    },
    {
        alias: 'candidates',
        path: '.claude/context/issues.xml',
        metaFiles: issuesFiles,
    },
]

export default definePlugin({
    name: 'rct-plugin-issue-scope',
    files,
    setup() {
        for (const f of files) {
            if (fs.existsSync(f.path)) continue
            const template = f.metaFiles?.find((mf: MetaFileEntry) => mf.alias.includes('template'))
            if (!template) continue
            const dir = path.dirname(f.path)
            if (!fs.existsSync(dir)) fs.mkdirSync(dir, { recursive: true })
            fs.copyFileSync(template.path, f.path)
        }
    },
})
```

**Why closure, not `this`:** Arrow function refactors and method extraction break `this` binding. Closing over the `files` array defined in the same scope is immune to binding surprises. This is the canonical pattern all plugin authors should follow.

### 4.2 Track-Work Plugin (`plugins/rct-plugin-track-work`)

**Current bug:** Unsafe `as keyof typeof templates` cast crashes on unknown alias.

**Redesign:** Same closure pattern as issue-scope — close over `files` array, guard the lookup:

```typescript
const files: FileEntry[] = [
    {
        alias: 'chores',
        path: '.claude/context/chores.xml',
        injectOn: 'SessionStart',
        metaFiles: [schema, templates.chores],
    },
    {
        alias: 'plans',
        path: '.claude/context/plans.xml',
        injectOn: 'SessionStart',
        metaFiles: [schema, templates.plans],
    },
]

export default definePlugin({
    name: 'rct-plugin-track-work',
    files,
    setup() {
        for (const f of files) {
            if (fs.existsSync(f.path)) continue
            const key = f.alias as keyof typeof templates
            if (!(key in templates)) continue
            const dir = path.dirname(f.path)
            if (!fs.existsSync(dir)) fs.mkdirSync(dir, { recursive: true })
            fs.copyFileSync(templates[key].path, f.path)
        }
    },
})
```

### 4.3 Tmux Plugin (`plugins/rct-plugin-tmux`)

#### 4.3.1 Factory Pattern Fix

The `createTool` factory unconditionally calls `preflight(params.target)` but not all tools have a `target` parameter. Two approaches:

**Option A (remove implicit preflight):** Each tool handler calls `preflight()` directly with the correct parameter. Factory just wraps registration.

**Option B (conditional preflight):** Factory checks if `target` exists in params before validating.

**Decision:** Option A. The factory's value is registration boilerplate reduction, not validation. Validation is a tool concern. This removes the leaky abstraction.

```typescript
// factory.ts — simplified
export function createTool<T extends z.ZodObject<z.ZodRawShape>>(
    options: ToolOptions<T>,
    handler: (params: z.infer<T>) => Promise<McpResult>,
) {
    const name = options.title.split(' ').map((s) => s.toLowerCase()).join('_')
    return {
        name,
        options,
        handler,
        register: (server: McpServer) =>
            server.registerTool<McpResultSchema, T>(name, options, handler as ToolCallback<T>),
    }
}
```

Each tool handler starts with:
```typescript
const check = await preflight(params.target)  // or preflight() for list
if (check) return check
```

#### 4.3.2 Error Message Fix (`read.ts`)

Use `stderr` in error messages, not `stdout`:
```typescript
if (exitCode !== 0) return err(`tmux capture-pane failed: ${stderr || stdout}`)
```

Apply same pattern to `split.ts` and `send.ts`.

#### 4.3.3 Plugin Export

Currently exports a minimal stub. Keep it minimal — tmux's value is the MCP server, not RCT pipeline integration. The plugin export exists so it appears in the registry.

```typescript
export default definePlugin({ name: 'rct-plugin-tmux' })
```

No `context()` or `trigger()` for v1.0.0. These are noted in the roadmap as potential v1.1 additions.

#### 4.3.4 Setup Command

`runSetup()` already has proper JSON shape validation (added in recent commits). Verify the `mcpServers` array guard is correct. Keep as-is if it passes review.

#### 4.3.5 InvalidTargetError Message

Update to reflect the actual regex acceptance:
```typescript
`Invalid tmux target '${target}'. Use session:window.pane format (e.g., dev:0.1).`
```

Keep the regex as-is (`/^[a-zA-Z0-9_:./$@%-]+$/`) — it's deliberately permissive for tmux compatibility. The error message should describe the common format, not enumerate all accepted patterns.
