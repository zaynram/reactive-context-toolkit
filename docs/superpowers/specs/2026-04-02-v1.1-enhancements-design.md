# v1.1 Enhancement Specification

**Date:** 2026-04-02
**Branch:** TBD (from main after v1.0.0 merge)
**Status:** Design complete, pending implementation
**Prerequisite:** v1.0.0 released and merged

## Motivation

Six issues discovered through real RCT consumption across four projects (daisy, claude-lib, tasktools, trust-generator) during the v1.0.0 rollout. All are grounded in actual workflow friction, not speculation.

## Changes

### 1. Plugin Path Overrides

**Problem:** Builtin plugins hardcode file paths (e.g., `.claude/context/chores.xml`). DAISY's files live at `dev/chores.xml`. Users must either move files or forgo plugins entirely.

**Solution:** Allow `globals.plugins` entries to be objects with path overrides:

```json
"plugins": [
    { "name": "rct-plugin-track-work", "paths": { "chores": "dev/chores.xml" } },
    "rct-plugin-issue-scope"
]
```

**Implementation:**

- Update `GlobalsConfig.plugins` type: `(string | PluginRef)[]` where `PluginRef = { name: string; paths?: Record<string, string> }`
- In `applyPlugins`, after resolving a plugin, if the entry is an object with `paths`, override matching `files[].path` by alias
- Update JSON schema (`rct.config.schema.json`)
- String entries are unchanged (backwards compatible)

**Files:** `src/config/types.ts`, `src/config/schema.ts`, `rct.config.schema.json`

### 2. Plugin `contextOn` Filter

**Problem:** Plugin `context()` fires on every hook invocation. No way to restrict to specific events.

**Solution:** Add optional `contextOn` property to `RCTPlugin`:

```typescript
interface RCTPlugin {
    contextOn?: HookEvent | HookEvent[] // defaults to all events if omitted
}
```

**Implementation:**

- In `src/cli/hook.ts` context loop, check `contextOn` before calling `context()`:
    ```typescript
    const contextEvents =
        ext.contextOn ?
            Array.isArray(ext.contextOn) ?
                ext.contextOn
            :   [ext.contextOn]
        :   null // null = all events
    if (contextEvents && !contextEvents.includes(event)) continue
    ```
- Thread `contextOn` from plugin through `applyPlugins` extensions
- Update `PluginExtensions.contexts` type to include `contextOn`

**Files:** `src/plugin/types.ts`, `src/config/schema.ts`, `src/cli/hook.ts`

### 3. Plugin Setup Idempotency — No Change

**Decision:** Setup already has internal guards (existsSync). Overhead is negligible. No code change. Document that plugins must make setup idempotent.

### 4. Meta `include: ["plugins"]`

**Problem:** `generateMeta` doesn't handle `include: ["plugins"]`.

**Solution:** When `plugins` is in the include list, emit plugin names with display names:

```xml
<plugins>
    <plugin name="track-work" ref="rct-plugin-track-work"/>
    <plugin name="issue-scope" ref="rct-plugin-issue-scope"/>
    <plugin name="scope-guard" ref="./.claude/hooks/rct/scope-guard.ts"/>
</plugins>
```

**Implementation:**

- `generateMeta` receives the resolved config which has `globals.plugins`
- For each plugin name, emit a `<plugin>` element with displayName and ref
- If format is JSON, emit as `{ plugins: [{ name, ref }] }`

**Files:** `src/engine/meta.ts`

### 5. `definePlugin` Returns New Object

**Problem:** `definePlugin` mutates `plugin.files` in-place, making the closure pattern fragile.

**Solution:** Return a new object with spread:

```typescript
export function definePlugin(plugin: RCTPlugin): RCTPlugin {
    if (!plugin.files) return plugin
    return {
        ...plugin,
        files: plugin.files.map((f) => ({
            ...f,
            path:
                path.isAbsolute(f.path) ?
                    f.path
                :   path.resolve(process.cwd(), f.path),
            metaFiles: f.metaFiles?.map((mf) => ({
                ...mf,
                path:
                    path.isAbsolute(mf.path) ?
                        mf.path
                    :   path.resolve(process.cwd(), mf.path),
            })),
        })),
    }
}
```

Non-breaking: consumers get the same resolved plugin object. The difference is the original `files` const is unmodified.

**Files:** `src/lib/plugin.ts`

### 6. Builtin Plugin Import — Use `import.meta.resolve`

**Problem:** `path.resolve(__dirname, '..', '..', 'plugins', ...)` is fragile.

**Solution:** Use `import.meta.resolve` (Bun-native):

```typescript
async function importBuiltin(name: string): Promise<RCTPlugin> {
    try {
        const { default: plugin } = await import(name)
        return plugin
    } catch {
        const resolved = import.meta.resolve(
            `../../plugins/${name}/src/index.ts`,
        )
        const { default: plugin } = await import(resolved)
        return plugin
    }
}
```

Remove the `path` import from `src/plugin/index.ts`.

**Files:** `src/plugin/index.ts`

## Plan Review Findings (addressed)

1. **Unmatched path override keys** — Emit warning when a `paths` key doesn't match any plugin file alias. Prevents silent misconfiguration.
2. **MetaFile path overrides** — Documented as out-of-scope. Top-level file paths only.
3. **No `triggerOn`** — Intentional. Triggers self-filter by returning `undefined`. Asymmetry with `contextOn` is acceptable.
4. **`import.meta.resolve` returns URL** — Bun's `import()` accepts file URLs. No issue.

## Non-Goals

- `triggerOn` filter (triggers self-filter, no demonstrated need)
- MetaFile path overrides (no current plugin needs this)
- Session-scoped state API for plugins (over-engineering for current needs)
- Plugin dependency resolution (no demonstrated need)
- Transform hooks (validated as unnecessary in assumption-surfacing)

## Test Plan

- Plugin path overrides: test that alias-matched paths are replaced
- contextOn: test that context only fires on matching events
- Meta plugins: test that plugin list appears in meta output
- definePlugin immutability: test that original files const is unchanged after definePlugin
- import.meta.resolve: verify builtins load in both workspace and GitHub installs

## Breaking Changes

None. All changes are additive or behavioral fixes. Existing configs and plugins work unchanged.
